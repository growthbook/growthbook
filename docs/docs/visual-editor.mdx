The GrowthBook Visual Editor is a Chrome side panel that lets you design A/B tests on a live website — change copy, swap images, tweak styles, even generate new variations with AI — and then ship those variations to production through your existing GrowthBook SDK. No code changes, no rebuilds, no separate staging site.

You can test virtually any change you can make to a page — and because the Visual Editor leads with AI, you don't have to think in CSS selectors or hand-edit code to get there. Describe the variation you want in plain English ("make the hero headline more urgent and the primary button green") and the editor generates the changes for you. That makes it uniquely approachable for product managers, marketers, designers, and anyone else who shouldn't have to write code to ship an experiment.

In particular, the Visual Editor is great for testing headlines and microcopy, hero images and product photography, layout, spacing, colors and typography, CTA placement and ordering — anything you can express as a DOM change.

When a variation grows beyond a handful of changes — a full page redesign, a multi-step flow rework, logic that depends on user data — those are usually better implemented in code and shipped as a [feature-flag experiment](/feature-flag-experiments). The Visual Editor shines for focused, surgical tests; heavyweight changes belong in your codebase.

:::warning Visual Editor may not work optimally on client-side rendered apps

With heavily client-side rendered apps (e.g., complex React, Vue, Svelte apps that re-render the page after hydration), the Visual Editor may not work optimally. Variations can flicker or be overwritten by re-renders. Consider using [Feature Flags](/app/features) instead for a better experience.

:::

## How it works

1. **Install** the [GrowthBook Visual Editor Extension](https://chrome.google.com/webstore/detail/FIX THIS) and open the side panel.
2. **Connect** it to your GrowthBook account with a Personal Access Token.
3. **Edit** your live site — describe a variation in plain English (AI mode), or point and click (Manual mode).
4. **Preview** with a single-click preview link, then publish through your existing SDK.

See [Install and connect](/app/visual/install-and-connect) for the full first-time setup.

## Requirements

- A Chromium-based browser (Chrome, Edge, Brave, Arc) with the [GrowthBook Visual Editor extension](https://chrome.google.com/webstore/detail/growthbook-devtools/FIX) installed.
- One of GrowthBook's client-side or Edge SDKs integrated into your site: [Script Tag](/lib/script-tag), [JavaScript](/lib/js), [ReactJS](/lib/react), or an [Edge SDK](/lib/edge/cloudflare).

### Allow Visual Experiments on the SDK Connection

Visual experiments are delivered to your site through an SDK Connection. The connection must be configured to include them:

1. In the GrowthBook UI, go to **SDK Configuration → SDK Connections**.
2. Open the connection your site uses.
3. Click **Edit** in the upper right.
4. Enable the **Include Visual Experiments** toggle.

<MaxWidthImage border>
![Enable Include Visual Experiments on your SDK Connection](/images/sdk-connection-visual-editor.png)
</MaxWidthImage>

If this toggle is off, visual experiments are stripped from the SDK payload and nothing will render — even if everything else is set up correctly. This is the single most common cause of "my variation isn't showing."

## Two ways to edit

The Visual Editor has two modes that you can mix and match within the same experiment.

### AI mode — describe the change

Type what you want in plain English ("change the hero headline to something more urgent", "make the primary button green and a bit bigger", "swap the hero image for one with a person using a laptop"). The AI proposes selector-grounded changes you can preview, accept, or edit.

AI mode is the fastest way to:

- Get a working draft of a variation from a rough idea
- Generate several headline candidates at once
- Produce new hero images with text-to-image generation
- Make multi-property changes ("make this card more premium") without touching each control

See [AI mode](/app/visual/ai-mode) for prompting tips, model selection, and how acceptance works.

:::info AI Mode

In order to enable AI mode, you must enable AI features from the GrowthBook Settings -> General -> AI Settings page.

:::

### Manual mode — point and click

Click any element on the page and you get a structured editor for its content, typography, background and border, spacing, and visibility. It's the right choice when you know exactly what you want to change and you want pixel-level control.

Manual mode includes:

- **Content** — text, inner HTML, attributes, link targets, CSS classes
- **Typography** — font size, weight, color, alignment, line-height, letter-spacing
- **Background & Border** — colors, images, borders, box shadow presets
- **Spacing** — a concentric padding/margin editor with drag-to-scrub and `auto` centering
- **Visibility** — opacity, display
- **Drag and drop** — reposition elements on the page
- **Global CSS & Custom JS** — escape hatches for anything the structured editors don't cover

See [Manual mode](/app/visual/manual-mode) for the full reference.

## Creating a Visual Experiment

To create a new visual experiment:

1. Navigate to the page you want to test in your browser.
2. Open the GrowthBook side panel and click **Create Experiment**.
3. Select the targeting URL rules, choose your randomization unit, add a hypothesis (optional), and click **Create**.
4. Now edit the page. Updates will automatically be saved.
5. Click "done" or return to the GrowthBook UI when you're ready to start the experiment.

### URL Targeting

GrowthBook needs to know which page(s) of your site the experiment should run on. URL targeting is required for the Visual Editor to work.

:::note Pass `url` as a targeting attribute

For URL targeting to apply at runtime, your SDK must include `url` as a targeting attribute. See [Targeting Attributes](/features/targeting).

:::

If your experiment runs on a single static page, enter the full URL (e.g., `https://www.example.com/pricing`).

If your experiment runs on dynamic URLs (e.g., all pages under `/post/`), click **Advanced Mode** and provide:

1. A single representative URL to load in the Visual Editor (e.g., `https://www.example.com/post/my-first-post`).
2. A targeting pattern to match all qualifying URLs (e.g., `/post/*`).

:::tip Simple URL targeting is recommended

Simple targeting covers nearly every real-world case and is much easier to get right than regex. Use regex only when you really need it.

:::

#### Simple Targeting

Simple URL targeting supports:

- Full URLs (e.g., `https://www.example.com/pricing`)
- Paths (e.g., `/pricing`)
- Query strings (e.g., `/pricing?utm_source=email`)
- Hashes/anchors (e.g., `/pricing#more-info`)
- Wildcards (e.g., `/posts/*` matches `/posts/123` and `/posts/2023/03/30/my-post`)
- Trailing-slash and protocol differences are ignored
- Extra query parameters are ignored (e.g., `/pricing?plan=pro` matches `/pricing?utm_source=email&plan=pro`)

#### Regex Targeting

Regex targeting accepts full regular expressions. Escape any special characters you don't intend to interpret.

```js
https?:\/\/(www\.)?example\.com\/pricing\/?
```

Or match on path only:

```js
^\/pricing\/(pro|enterprise)
```

## Previewing and QA

While an experiment is still a draft, you can preview each variation by appending a querystring to your URL:

```
https://www.example.com/pricing?my-experiment-id=1
```

`0` is the control, `1` is the first variation, and so on. The experiment ID is shown on the right side of the experiment page in GrowthBook under **Settings**.

:::note Draft preview requires the "Include Drafts" toggle

Drafts are gated on the SDK Connection's **Include Drafts** toggle. Until that toggle is on, draft preview links return the control.

:::

See [Preview and QA](/app/visual/preview-and-qa) for the debug panel, the live-experiment editing warning, and other QA tooling.

## Deploying to Production

To deliver your variations to real users, you need a GrowthBook SDK installed on your site. The Visual Editor works with the [Script Tag](/lib/script-tag), [JavaScript](/lib/js), [React](/lib/react), and [Edge SDKs](/lib/edge/cloudflare). The Script Tag SDK is the quickest path; the Edge SDK is the best for performance because it eliminates flicker entirely.

## Stopping an Experiment

When your experiment is finished, click **Stop Experiment** at the top of the results page. You'll be prompted for the reason, the conclusion, and which variation (if any) won.

If a variation won, you can optionally enable a **Temporary Rollout** when stopping. This continues running the experiment with the same targeting conditions, but sends 100% of traffic to the winning variation and disables the `trackingCallback`.

It's called a **temporary** rollout because you shouldn't rely on the SDK to deliver the winning variation forever. It's best practice to have your engineering team re-implement the changes directly in your site's code, because:

1. Changes implemented in code render faster — your site loads quicker.
2. Code changes are picked up by SEO crawlers.
3. Visual Editor changes require the SDK to download config from GrowthBook on each page load. They're lightweight individually, but they accumulate.
4. Reduce the chance of conflicts. If two visual editor experiments try to change the same element at the same time, the result may not be what you expect. Moving the winning variation to code eliminates the risk.

## Change list

The bottom of the side panel shows a collapsible **Changes** panel that lists every change in the current variation:

- Each row shows the change kind (text, style, image, layout, etc.), the target selector, and a one-line summary.
- Hover a row to highlight the affected element on the page.
- Click a row to jump back to that element with the relevant property group expanded.
- Each row has a **Revert** action to drop just that change while keeping the rest of the variation.

Reverting an image change also clears any related `srcset` and `<picture>`-source removals the editor added when applying the swap, so the original image is fully restored.

## Switching variations

At the top of the sidebar is a variation switcher. Switch between control, variant 1, variant 2, etc. The Changes panel updates to show changes for the active variation.

- The **control** is read-only — you can browse changes (there shouldn't be any) but not edit.
- Use the **+** button next to the variation tabs to add a new variant to the experiment.

## Finishing up

Changes to your experiment are automatically saved to GrowthBook. When you're satisfied with the variants, click **Done** in the footer to return to GrowthBook. Once there you can manually edit the rules, and start/stop the experiment.
