A Ramp Schedule rolls a feature out in stages instead of switching it on for everyone at once. You define the stages up front, and GrowthBook moves through them on a schedule you set. A typical schedule might show the feature to 5% of users, then 25%, then 100%. Between stages, the rollout can wait a set amount of time or pause for a teammate's approval, so you widen exposure gradually and can stop early if something looks off.

## How ramp schedules work

A Ramp Schedule attaches to a single **Targeting rule** (a rule that serves a value to matching users, with optional targeting conditions). It is a release plan on top of that rule, not a separate rule type. You add one from the rule editor by choosing **Ramp-up** as the release plan.

The schedule drives the rule through an ordered list of **steps**. Each step only sets the fields you want to change; everything else carries over from the previous step. A step can change:

- **Coverage**: the percentage of matching users who get the rule's value.
- **Targeting**: the condition, saved groups, and prerequisites.
- **Environments**: which environments the rule applies to.
- **Value**: the value the rule serves.

A step advances when its gates clear:

- **Time gate**: a hold duration before the step is eligible to advance. Leave it empty for a step that waits on approval instead.
- **Approval**: when a step requires approval, a team member must approve it in the UI or API. Approval is always the final gate, so any time or sample-size conditions clear first.
- **Minimum sample size**: hold until a set number of users have been exposed. This applies only to [monitored steps](#monitored-steps-and-guardrails).

When every gate clears, GrowthBook applies the step's patch to the rule and starts the next step. Each advance publishes a new feature revision automatically, so your SDKs pick up the change through the normal feature payload.

### Start, end, and cutoff

- **Start date**: leave it empty to start as soon as the rule is published. Set a future date to delay activation; the rule stays disabled until then.
- **End actions**: applied after the final step, on top of everything the steps have already changed. Use these to set the rule's final state, such as 100% coverage with targeting removed.
- **Cutoff date**: a hard kill date. When it is reached, the schedule completes and the rule is disabled, regardless of step progress. Use this for time-boxed rules.

### Lockdown

Set the schedule to **locked** to block other feature edits while it is actively running. This prevents accidental changes from competing with the schedule.

:::note
Lockdown only applies while the schedule's status is `running`. It does not lock the ramp's own progression (use pause for that), and it never blocks the changes the ramp engine makes as it advances.
:::

## Ramp schedule vs. Safe Rollout

A [Safe Rollout](/features/safe-rollouts) is a Ramp Schedule with guardrail monitoring turned on. Both use the same engine. The difference is whether you watch metrics as the ramp progresses.

- Use a plain **Ramp Schedule** when you want a release plan that runs on its own: step coverage up over time, broaden targeting in stages, gate jumps behind approval, or stop at a cutoff date.
- Use a **monitored** Ramp Schedule (a Safe Rollout) when you also want GrowthBook to watch guardrail metrics and automatically hold or roll back if the release harms them.

You configure monitoring per step, so a single schedule can mix unmonitored and monitored steps.

## Adding a ramp schedule

### 1. Add a Targeting rule

When you add a rule to a feature, choose **Targeting rule** as the rule type.

<MaxWidthImage border>
![Rule type picker showing Targeting rule, Experiment, and Bandit, with a callout pointing to Safe Rollouts](/images/features/rule-type-picker.webp)
</MaxWidthImage>

:::tip
Looking for Safe Rollouts? Choose a Targeting rule and turn on guardrail monitoring in its ramp-up schedule. The **Show me** shortcut on the rule type screen configures a fully monitored ramp for you.
:::

### 2. Choose the Ramp-up release plan

In the rule settings, set the release plan to **Ramp-up** (or **Monitored Ramp-up** to watch guardrail metrics). The ramp editor opens so you can define the steps.

### 3. Configure the steps

Start from a preset or template, or build steps by hand. For each step, set the coverage and the hold behavior (a time interval, an approval gate, or both). Add as many steps as you need.

While a ramp is running, the rule's coverage is controlled by the schedule. Pause or end the ramp to make immediate changes.

<MaxWidthImage border>
![A running ramp schedule showing the current step and the timeline of upcoming steps](/images/features/ramp-schedule-timeline.webp)
</MaxWidthImage>

### 4. Set the start and cutoff

Optionally set a start date to delay activation and a cutoff date to disable the rule at a fixed time.

### 5. Publish

Publish the revision to arm the schedule. A schedule configured inside a draft stays in the `pending` state until that draft is published.

## Monitored steps and guardrails

Mark a step as **monitored** to run guardrail analysis while it is active. GrowthBook splits enrolled users 50/50 between the new value and the existing value and analyzes your guardrail metrics, using the same engine as a [Safe Rollout](/features/safe-rollouts).

On a monitored step, `coverage` is the share of users who get the new value, and an equal-sized control group gets the existing value. Coverage is capped at 0.5: at that point the new value and the control each reach half your users, so no one is left out.

:::note
A monitored step with `coverage: 0.5` is a full 50/50 split: 50% of users get the new value and 50% get the existing value. Set `coverage: 0.25` to show the new value to 25% of users.
:::

Configure monitoring once on the schedule:

- **Data source and exposure query**: where traffic and metric data come from.
- **Guardrail metrics**: at least one metric to watch for regressions.
- **Signal metrics**: additional metrics to watch. A failing signal metric can hold the ramp but never rolls it back.
- **Update schedule**: how often snapshots refresh, in `auto` or `manual` mode.
- **Health actions**: what to do when a check fails.

You set a health action for each type of problem:

| Trigger                    | Options                    |
| -------------------------- | -------------------------- |
| Guardrail regression       | `rollback`, `hold`, `warn` |
| Sample ratio mismatch      | `rollback`, `hold`, `warn` |
| Multiple exposures         | `rollback`, `hold`, `warn` |
| No traffic after the grace period | `rollback`, `hold`, `warn` |

A `rollback` rewinds the rule to its pre-ramp state and ends the schedule. A `hold` pauses the ramp for review. A `warn` surfaces the issue without stopping the ramp. The no-traffic grace period defaults to 24 hours.

:::note
Sticky bucketing is disabled on monitored steps, because the hash ranges shift as coverage grows. Minimum sample size applies only to monitored steps, since unmonitored steps have no analysis to evaluate.
:::

## Operating a running schedule

Once started, a schedule reports its status on the feature's rule.

| Status        | Meaning                                                          |
| ------------- | ---------------------------------------------------------------- |
| `pending`     | Created but not yet armed. Publish the revision to start.        |
| `ready`       | Armed and waiting for the start date.                            |
| `running`     | Actively advancing through steps.                                |
| `paused`      | Halted by a user or a `hold` health action. Resume to continue.  |
| `completed`   | All steps applied and end actions fired.                         |
| `rolled-back` | Returned to the pre-ramp state.                                  |

From the UI or API you can start, pause, resume, manually advance, jump to a specific step, approve a gated step, roll back to the pre-ramp state, or restart a finished schedule. Every transition is recorded in the schedule's event history.

## Templates

Save a schedule's steps and end actions as a **Ramp Schedule Template** to reuse them across features. Apply a template when you create a schedule to inherit its steps, and override any of them as needed.

## Managing via the API

Ramp Schedules have a full REST API under `/ramp-schedules`. This example attaches a three-step ramp to an existing rule, with the final step gated on approval:

```bash
curl -X POST https://api.growthbook.io/api/v1/ramp-schedules \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Checkout v2 rollout",
    "featureId": "checkout-v2",
    "ruleId": "fr_abc123",
    "steps": [
      { "interval": 3600, "actions": [{ "patch": { "coverage": 0.1 } }] },
      { "interval": 86400, "actions": [{ "patch": { "coverage": 0.5 } }] },
      { "interval": null, "holdConditions": { "requiresApproval": true },
        "actions": [{ "patch": { "coverage": 1.0 } }] }
    ]
  }'
```

When you pass both `featureId` and `ruleId`, GrowthBook fills in the target on every action, so you only supply the fields you want to change. See the [API reference](/api) for the full set of endpoints and lifecycle actions.
