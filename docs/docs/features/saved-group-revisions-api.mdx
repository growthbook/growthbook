# Saved Group Revisions API

:::info BETA
The saved group revisions API is currently in beta. Endpoint shapes may change in minor ways before GA.
:::

This guide walks through editing a saved group through GrowthBook's revision-aware REST API. Every change goes through the same draft → review → publish lifecycle that the [UI uses](/features/publishing-and-approval-flows#saved-groups), so a draft you start via REST can be reviewed in the UI (and vice versa).

If you only need to read or replace a saved group atomically — and your org does not require approvals — you can keep using [`POST /saved-groups/{id}`](/api/#operation/updateSavedGroup) and the `bypassApproval: true` flag. The endpoints below are for callers who need to stage changes, run approvals, or coordinate with the UI revision flow.

The full endpoint reference is in the [REST API docs](/api/#tag/saved-group-revisions). This page shows the lifecycle and the request shapes you'll typically use.

## Concepts

A **saved group** has one live revision and zero or more open **draft revisions**. A draft holds a set of proposed changes (JSON Patch ops) against a snapshot of the saved group taken when the draft was opened.

| Status              | Meaning                                                                         |
| ------------------- | ------------------------------------------------------------------------------- |
| `draft`             | Open, editable. Authors can keep staging changes.                               |
| `pending-review`    | Submitted for review. Reviewers can approve, comment, or request changes.       |
| `changes-requested` | A reviewer asked for changes. Edit the draft and re-request review.             |
| `approved`          | Approved and ready to publish.                                                  |
| `merged`            | Published. The proposed changes are now part of the live saved group. Terminal. |
| `discarded`         | Abandoned. Terminal.                                                            |

When the org requires approvals for saved groups (see [Approval Flows](/features/publishing-and-approval-flows#saved-group-approval-flows)), publishing is blocked unless the revision is `approved`, the caller has the `bypassApprovalChecks` permission on every target project, or the org has enabled **REST API always bypasses approval requirements**.

## Lifecycle in one example

Below is the most common flow: stage a values change on an existing list saved group, request review, get it approved, and publish.

### 1. Create a draft

```bash
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"title": "Add Q2 beta cohort"}'
```

The response contains the new revision, including its integer `version`:

```json
{
  "revision": {
    "id": "rev_...",
    "version": 4,
    "status": "draft",
    "baseSavedGroup": { "...": "snapshot of the live saved group" },
    "proposedSavedGroup": { "...": "snapshot + proposed changes applied" },
    "proposedChanges": []
  }
}
```

You'll use `version` (here, `4`) on every subsequent endpoint.

### 2. Stage the change

Append items to a `list` saved group's `values`:

```bash
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/items/add' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"items": ["user_42", "user_77"]}'
```

`POST .../items/add` and `POST .../items/remove` are idempotent on individual items (duplicates are deduped, missing removals are no-ops) and stack on top of the current draft state, so multiple calls accumulate.

For an atomic full replacement, use `PUT .../values` instead — but only when you know no other writer is competing.

### 3. Request review

```bash
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/request-review' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{}'
```

The revision moves from `draft` to `pending-review` and reviewers are notified per org settings.

### 4. Submit a review

A different user (or the same user calling `request-changes` / `comment`) submits a decision. With `blockSelfApproval` enabled, authors and contributors cannot approve their own drafts.

```bash
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/submit-review' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"decision": "approve", "comment": "LGTM"}'
```

`decision` is one of `approve`, `request-changes`, or `comment`.

### 5. Publish

```bash
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/publish' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{}'
```

After publish, the revision's status becomes `merged` and the proposed changes are applied to the live saved group.

## Auto-creating a draft on edit

Every field-edit endpoint (`PUT .../metadata`, `PUT .../condition`, `PUT .../values`, `PUT .../archive`, `POST .../items/add`, `POST .../items/remove`) accepts the literal string `"new"` in place of a version number. This opens a fresh draft, applies the edit, and returns the new revision in one round trip.

```bash
curl -X PUT 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/new/metadata' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{
    "revisionTitle": "Rename to internal-beta",
    "name": "internal-beta",
    "description": "Cohort moved to the internal-only beta program"
  }'
```

`revisionTitle` and `revisionComment` are only applied when a new draft is created — they're ignored when editing an existing draft.

## Reading revisions

| Endpoint                                                  | Use it for                                                                                                       |
| --------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `GET /saved-groups/revisions`                             | List revisions across every saved group. Filter by `savedGroupId`, `status`, `author`, or `mine=true`.           |
| `GET /saved-groups/{id}/revisions`                        | List revisions for one saved group.                                                                              |
| `GET /saved-groups/{id}/revisions/latest`                 | The most recently updated open draft. 404 if no open draft. `mine=true` filters to drafts you authored.          |
| `GET /saved-groups/{id}/revisions/{version}`              | Fetch a specific revision, including its `baseSavedGroup`, `proposedSavedGroup`, `reviews`, and `activityLog`.   |
| `GET /saved-groups/{id}/revisions/{version}/merge-status` | Dry-run merge against the current live state. Returns `hasConflicts`, `canAutoMerge`, and per-field `conflicts`. |

`baseSavedGroup` is the snapshot taken when the draft was opened. `proposedSavedGroup` is what the live saved group would look like if the draft were merged right now — useful for previewing without interpreting `proposedChanges` (raw JSON Patch ops).

## Handling merge conflicts

When two drafts touch the same field, or someone publishes a change while your draft is open, the second `publish` fails with `409 Conflict`. The response body includes the conflicting fields.

The recovery flow is to rebase, then re-publish:

```bash
# 1. Inspect conflicts
curl 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/merge-status' \
  -H 'Authorization: Bearer YOUR_API_KEY'

# 2. Rebase, picking a resolution strategy per conflicting field
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/rebase' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{
    "conflictResolutions": {
      "values": "union",
      "description": "overwrite"
    }
  }'

# 3. Retry publish
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/4/publish' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{}'
```

Resolution strategies, per conflicting field:

- `overwrite` — keep the draft's value.
- `discard` — keep the live value.
- `union` — concatenate arrays (only valid for `values` on list saved groups). Pass `customValues` to supply your own merged array instead of the default union.

`merge-status` is purely informational — it doesn't lock anything. If you need strict optimistic locking, call `merge-status`, then `publish`, and on `409` re-fetch and try again.

## Reverting

`POST /saved-groups/{id}/revisions/{version}/revert` creates a new revision whose content matches the specified historical revision. Pass `{"strategy": "draft"}` to stage the revert as a draft (default) or `{"strategy": "publish"}` to publish immediately. The publish path obeys the same approval rules as `publish`.

```bash
curl -X POST 'https://api.growthbook.io/api/v1/saved-groups/grp_abc123/revisions/2/revert' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"strategy": "draft", "title": "Roll back the Q2 cohort change"}'
```

## Approval bypass

<CommercialFeature feature="require-approvals" />

When approvals are required, three paths can publish without an `approved` revision:

1. **Per-call bypass** — `POST /saved-groups` and `POST /saved-groups/{id}` accept `bypassApproval: true`. The caller must hold `bypassApprovalChecks` on every target project. This path skips the revision flow entirely and writes directly to the live saved group.
2. **Per-user bypass on publish** — when calling `POST .../revisions/{version}/publish`, the publish succeeds if the caller has `bypassApprovalChecks` on the saved group's projects, even if the revision is not `approved`.
3. **Org-wide bypass** — admins can enable **REST API always bypasses approval requirements** in org settings, after which every REST publish acts as a bypass regardless of the caller's permission.

## Permissions

The endpoints inherit the saved group's existing permission gates:

- Read endpoints require `canReadSavedGroup` on the saved group's projects.
- Edit, request-review, rebase, revert, and publish endpoints require `canUpdateSavedGroup`. Publish additionally re-checks against the live saved group at the moment of publish so a `projects` move can't launder write access.
- Submitting a review uses the same write gate as editing the saved group.
